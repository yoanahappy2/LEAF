LEAF: 面向低资源语言翻译的分层评估与保证框架

发现并缓解 ReAct 检索系统中的查询漂移问题

黄咏郁（2026110008）

大模型驱动的软件开发 · 清华大学 · 2026年6月

摘要

全球 7,000 种语言中超过 40% 面临濒危，主流大语言模型（GPT-4、GLM-4）对低资源语言的零样本翻译 BLEU 得分为 0。本文提出 LEAF（Layered Evaluation and Assurance Framework），一个面向低资源语言翻译品质保证的 Multi-Agent 框架，以排湾语（Paiwan, ISO 639-3: pag）为验证语言。

我们在 ReAct 架构中发现了一种稳定的失败模式——查询漂移（Query Drift）：LLM 在迭代推理中擅自改写检索查询，漂移率高达 35%，其中 71% 直接导致翻译错误。LEAF 通过四层品质保证（CoreVocab / RAG / Grammar / Verify）和 Multi-Agent 编排，将漂移率从 35% 降至 0%，翻译准确率从 65% 提升至 95%。消融实验证明，30pp 提升全部来自架构设计本身。

实验在排湾语（5,923 条语料）和阿美语（6,413 条语料）上完成双语言交叉验证。关键词：低资源语言翻译、Multi-Agent、ReAct、查询漂移、消融实验

1 引言

1.1 问题背景

低资源语言面临双重困境：数字语料匮乏导致大语言模型无法直接处理，而语言濒危的紧迫性要求技术方案"少数据、快部署"。主流 LLM 的零样本翻译 BLEU 为 0.0（表 1）。

表 1：零样本基线实验

即使 20-shot ICL，BLEU 仍为 0——排湾语在 LLM 预训练数据中几乎不存在。

1.2 RAG 的希望与脆弱性

仅 242 条语料，关键词 RAG 即可将完全匹配率提升至 90%（表 2）。

然而，RAG 在低资源场景中极其脆弱——语料太少，容不下模糊匹配。当 LLM 在 ReAct 循环中改写查询时（如"月亮"→"月亮的族语怎么说"），精确匹配立刻失败。

1.3 贡献

  1. 量化查询漂移：ReAct 单 Agent 中漂移率 35%，71% 直接导致错误
  2. 提出 LEAF 框架：四层品质保证 + Multi-Agent 编排，架构级约束查询透传
  3. 完整消融验证：8 配置 Truth Table + 双语言交叉验证

2 相关工作

低资源语言 NLP。传统统计机器翻译需大量平行语料（Koehn et al., 2007），零样本跨语言迁移（Conneau et al., 2020）依赖类型学相似性。

RAG 在翻译中的应用。检索增强生成（Lewis et al., 2020）补充 LLM 知识盲区，但现有工作未关注查询被改写导致的失败。

Multi-Agent 系统。AutoGen（Wu et al., 2023）等关注任务分解，较少研究编排对检索品质的影响。

ReAct 范式。Yao et al. (2023) 让 LLM 在推理中调用工具，正是此"自由推理"引发了查询漂移。

3 方法

3.1 系统架构

LEAF 包含 3 个核心 Agent：

  • Coordinator Agent（协调智能体）：系统协调者，负责 ReAct 流程控制与工具路由。核心职责是保护用户原始 Query——无论后续进行多少轮推理或工具调用，原始问题都会被完整保留，不被 LLM 自动改写。
  • Knowledge Agent（知识智能体）：封装所有语言知识来源，包括 Exact Match、RAG Search 和 CoreVocab 词汇库，负责找到候选翻译结果。
  • Quality Agent（品质智能体）：通过反向翻译验证检查候选答案是否正确。验证失败则触发下一轮检索与修正。

简言之：Knowledge Agent 负责「找得到」，Quality Agent 负责「找对了」，Coordinator Agent 负责确保整个流程不会因 Query Drift 而偏离用户意图。

3.2 四层品质保证

层级名称功能机制
L1CoreVocab词汇约束词典 + 精确匹配，输出约束在已知词汇空间
L2RAG检索增强关键词 + 向量（FAISS）混合检索
L3Grammar语法注入音韵规则 + 缀词分析
L4Verify反向验证反向翻译检查是否回到原文

3.3 查询漂移定义

ReAct 单 Agent 中 LLM 会改写检索查询。漂移类型：扩展型（月亮→月亮的族语怎么说）、改写型（孩子→小孩子的怎么说）、语义漂移型（朋友→朋友说什么语言）。

3.4 架构级约束

核心不是 prompt engineering，而是架构设计：① 查询透传（代码层面确保原封不动）② 结构化工具调用（schema 约束参数）③ 失败回退（切换检索策略而非改写查询）。

4 实验

4.1 实验设置

排湾语（5,923 条）+ 阿美语（6,413 条）双语言验证。基座模型 GLM-4-Flash。测试集 20 题中文→排湾语。指标：BLEU、Exact Match、漂移率。

5 讨论

查询漂移是 LLM 内在行为，Constrained Prompt 仅 35%→30%。LEAF 通过职责分离、结构化传递、失败恢复三项架构设计解决。双语言验证中 CoreVocab 贡献一致（-33pp vs -35pp），捕捉到低资源翻译通用瓶颈。局限：测试集 20 题规模较小；GLM-4-Flash 上 L4 效果不稳定；仅验证单向翻译。

6 结论

  1. 查询漂移是 ReAct 固有缺陷：35% 漂移率，71% 致错，prompt engineering 无法根治
  2. 架构级约束是方案：Multi-Agent 编排降至 0% 漂移，65%→95%（+30pp），全部来自架构设计
  3. 框架可迁移：排湾语+阿美语双语言验证一致

核心洞察：架构设计优于 prompt engineering。
