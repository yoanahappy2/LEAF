## 一、系統架構概覽

| 組件 | 說明 |
|------|------|

## 二、實驗 1：翻譯品質四層消融（雙語言驗證）

> 驗證翻譯引擎中各組件的貢獻。每語言 50 條測試，klokah 交叉驗證。

| 實驗組 | 排灣語 | 降幅 | 阿美語 | 降幅 |
|--------|--------|------|--------|------|
| Control（4 層全開） | **79.2%** | — | **73.5%** | — |
| 關閉 CoreVocab + ExactMatch | 45.8% | **-33.4pp** | 38.8% | **-34.7pp** |
| 關閉 Grammar 語法注入 | 75.0% | -4.2pp | 73.5% | +0.0pp |
| 關閉往返驗證 | 83.3% | +4.1pp | 73.5% | +0.0pp |

**結論：**
- CoreVocab + ExactMatch 是系統的絕對支柱（關掉降 33-35pp，兩個語言一致）
- Grammar 注入對有語法書的語言有效（排灣語 -4.2pp），對無語法書的語言無影響（阿美語 0pp）
- 往返驗證在 GLM-4-flash 上有反效果（反向翻譯品質太差），建議用更強模型時才開啟
- **可遷移性驗證成功**：兩語言 Control 只差 5.7pp（79.2% vs 73.5%）

## 三、實驗 2：Multi-Agent 消融（單任務對比）

> 15 個標準測試場景（翻譯/知識/教學/品質/對話），對比 4 種 Agent 配置。

| 配置 | 總通過率 | Token 消耗 |
|------|---------|-----------|
| 4 Agent 全開（Control） | **93%**（14/15） | 140,554 |
| 關掉 Quality Agent | 93%（14/15） | 127,550 |
| 關掉 Teaching Agent | 93%（14/15） | 148,192 |
| **單 Agent（無 RAG 工具）** | **47%**（7/15） | 7,134 |

### 按任務類型通過率

| 任務類型 | 4 Agent | No Quality | No Teaching | 單 Agent |
|---------|---------|-----------|-------------|---------|
| 翻譯 | **100%** | 100% | 100% | **0%** |
| 知識查詢 | 67% | 67% | 67% | 33% |
| 教學 | **100%** | 100% | 100% | 67% |
| 品質審核 | 100% | 100% | 100% | 100% |
| 閒聊 | 100% | 100% | 100% | 100% |

**結論：**
- **翻譯類 100% → 0%**：單 Agent 沒有 Knowledge Agent 的 RAG + Exact Match，完全無法翻譯排灣語。這是 Multi-Agent 分工的核心價值——專業任務交給專門的 Agent。
- Quality / Teaching 在此測試中對通過率影響不大，因其價值更多體現在互動品質而非是否通過
- 單 Agent Token 消耗極低（7,134），但代價是完全無法處理需要工具的任務

## 四、實驗 3：複合任務對比（Multi-Agent 價值證明）

> 4 個複合教學任務（教詞 + 出題 + 審核），對比 4 種系統配置。

### 綜合表現

| 配置 | 綜合分 | 詞彙覆蓋率 | Token 消耗 |
|------|--------|-----------|-----------|
| Full System（4 Agent） | 0.38 | 32% | 66,598 |
| No Plan（純 ReAct） | **0.20** | **12%** | 66,184 |
| No Self-Judge | 0.54 | 64% | 60,192 |
| Single LLM + RAG | 0.49 | 50% | ~15,000 |

### 按任務對比

| 任務 | Full System | No Plan | No Judge | Single LLM |
|------|------------|---------|----------|-----------|
| M01：問候語教學+測驗 | 0.27 | 0.21 | **0.69** | 0.60 |
| M02：數字教學+發音審核 | 0.31 | 0.28 | **0.75** | **0.75** |
| M03：家族稱謂教學+翻譯 | 0.15 | 0.15 | 0.30 | **0.52** |
| M04：自然詞彙+文化+填充 | **0.80** | 0.15 | 0.42 | 0.10 |

### 分析

**Plan 的價值（Full vs No Plan）：**
- No Plan 在 M04 只有 0.15（vs Full 0.80），差距 5 倍以上
- 沒有 Plan 時，LLM 無法自主組織「教詞→解釋→出題」的多步流程
- 綜合分 0.38 vs 0.20，Plan-Driven 模式提升近一倍

**Single LLM 的局限：**
- M02（數字）和 M03（家族）表現不錯，因為 RAG 直接找到了答案
- M04（自然詞彙+文化解釋+填充題）只有 0.10，因為單 LLM 無法同時處理翻譯+教學+出題三個子任務
- Full System 在 M04 的 0.80 遠超 Single LLM 的 0.10，說明 Multi-Agent 分工在複合任務中有明確優勢

**已知限制：**
- Full System 和 No Judge 的反轉（0.38 < 0.54）主要是 GLM-4.5-Air 的 tool calling 不穩定造成
- 當 Orchestrator 正確調用 Agent 時（如 M04），Full System 表現最好；但 tool calling 失敗時（如 M01-M03 的格式洩漏），品質反而不如不調工具
- **這是基座模型能力問題，不是框架設計問題**。用更強模型（GPT-4 / Claude）時，Full System 應穩定優於其他配置

## 五、實驗 4：長程自主運行

> MainLoop 自主運行，目標「排灣語初學者完整課程」，持續直到 Token 預算耗盡。

| 指標 | 數值 |
|------|------|
| 總 Loops | **59** |
| LLM 調用 | 436 次 |
| Token 消耗 | 605,192 |
| 運行時間 | 25 分鐘 |
| 連續失敗 | **0**（全程健康） |
| 停止原因 | Token 預算耗盡（正常） |

### 運行模式分佈

| 階段 | Loops | 說明 |
|------|-------|------|
| Plan-Driven | 3 | Teaching Agent 生成 3 步學習計畫並逐步執行 |
| ReAct | 54 | Plan 完成後，Orchestrator 自主決定每步學什麼 |
| Self-Judge | 每 5 輪 | Quality Agent 評估目標達成度，信心過低時調整策略 |

### 關鍵觀察

1. **Plan → ReAct 無縫切換**：前 3 輪按計畫走（認識問候語→練習發音→測驗），之後自動切到 ReAct 自主探索
2. **零失敗**：59 輪全部成功完成，連續失敗計數為 0
3. **Self-Judge 正常觸發**：每 5 輪調用 Quality Agent 評估，判斷「未達成」後繼續執行
4. **Checkpoint 持續保存**：每輪自動存 checkpoint，可隨時恢復運行
5. **Token 效率**：平均每輪 ~10,200 tokens，每輪包含 Orchestrator 決策 + Agent 調用 + 結果彙整
